Spring AI Alibaba 是阿里巴巴开源的企业级 AI 应用开发框架，属于 Spring AI 生态的一部分。它面向 Java / Spring 技术栈，重点解决模型接入、工具调用、工作流编排、RAG、多 Agent 协作和可观测性等问题。

如果从 Java 后端开发者的视角看，它的价值不只是“能调大模型”，而是把 AI 应用里常见的工程问题接进 Spring 体系。

这篇笔记来自我本地拆出来的一组 Spring AI Alibaba 学习 Demo，按模块分别验证了 ChatClient、RAG、MCP Server、Graph 编排、Ollama、本地图像语音能力、Remote RAG 和 Agent。文章不展开完整代码，只记录我在拆模块时觉得更值得沉淀的工程关注点。

## 它主要解决什么问题

一个 AI 应用落地时，通常不止一次模型调用。

真实工程里还会遇到这些问题：

- 模型供应商如何接入
- Prompt 和模型参数如何管理
- 工具调用如何注册和执行
- RAG 如何接入向量存储
- 工作流如何编排和持久化
- 多 Agent 如何通信
- 调用链路如何观测
- API Key、限流、重试如何处理

Spring AI Alibaba 的定位，就是在 Spring Boot / Spring Cloud 生态里，把这些能力整理成更容易落地的组件。

## 核心模块可以这样看

从能力上可以分成几类：

```text
Spring AI Alibaba
├── 模型接入
│   ├── DashScope Chat
│   ├── Embedding
│   ├── 图像生成
│   └── 语音能力
├── 应用编排
│   ├── Graph 工作流
│   ├── Tool Calling
│   ├── RAG
│   └── A2A Agent
├── 外部能力接入
│   ├── MCP Router
│   └── 自定义工具函数
└── 工程能力
    ├── 可观测性
    ├── 检查点
    ├── 重试
    └── 配置管理
```

这套结构比较适合 Java 团队，因为它没有要求你完全换一套技术栈，而是沿着 Spring 的方式接入。

## 我自己的 Demo 拆分方式

学习时我没有把所有能力塞进一个工程，而是拆成几个小模块：

| 模块 | 主要验证内容 |
|---|---|
| Chat Client | 基础对话、Advisor、对话记忆 |
| RAG | 向量存储、检索增强、工具调用 |
| MCP Server | 把本地工具暴露为 MCP 能力 |
| Graph Simple | 基于图的节点编排和条件流转 |
| Ollama | 本地模型接入 |
| Image / Audio | 图像生成、文本转语音等多模态能力 |
| Remote RAG | 远程知识库 / 检索链路接入 |
| Agent | 面向任务的 Agent 调用入口 |

这样拆的好处是每个模块都能单独运行和排查。AI 应用里的问题经常不是一个点坏了，而是模型、Prompt、工具、向量库、网络和配置一起影响结果。拆小后更容易判断问题到底在哪一层。

## 模型接入：先把 Chat Model 标准化

最基础的是模型接入。通过 starter 可以把 DashScope 等模型作为 Spring Bean 使用。

示例配置大概是这样：

```yaml
spring:
  ai:
    dashscope:
      api-key: ${DASHSCOPE_API_KEY}
      chat:
        enabled: true
        options:
          model: qwen-plus
          temperature: 0.8
          max-tokens: 2000
```

工程上要注意的是，不要把 API Key 写死在配置文件里。更适合的做法是走环境变量、密钥管理或部署平台的 Secret。

如果团队未来要接多个模型，也要尽早考虑：

- base URL 是否可配置
- 模型名是否可按环境切换
- 调用日志是否统一记录
- 失败后是否有降级或重试策略

这些问题不处理，后面从单模型切到多模型时会比较痛。

## Tool Calling：让模型调用业务能力

Tool Calling 是 AI 应用从“聊天”进入“办事”的关键。

Java 里常见做法是把一个业务函数注册成工具，让模型在需要时选择调用。比如天气查询、订单查询、库存检查、工单创建等。

工具设计时不要只关注“能不能调用”，还要关注：

- 工具描述是否清楚
- 参数是否足够结构化
- 参数是否有校验
- 错误结果是否能指导模型修正
- 是否需要人工确认

一个工具如果返回一大段模糊异常，模型很难继续处理。更好的做法是返回结构化错误，比如错误码、原因、建议动作。

## Graph 工作流：适合有明确流程的场景

不是所有 AI 应用都需要完全自主的 Agent。

很多企业场景反而更适合工作流：路径大体可控，只在某些节点使用模型判断。

比如客户反馈处理：

1. 先判断正面还是负面
2. 负面反馈继续分类
3. 按分类进入不同处理节点
4. 记录结果或派发工单

这种场景适合用图编排。代码里可以把每个节点拆开，再通过边和条件路由组织流程。

工作流的优点是可控、可调试、方便加检查点。缺点是灵活性不如完全自主 Agent。我的理解是：业务流程越明确，越应该优先考虑 Workflow，而不是一上来就做 Agent。

## RAG：重点不只是向量检索

RAG 看起来是“文档切片 + 向量检索 + Prompt 拼接”，但工程难点通常不在这几个词上。

更容易出问题的是：

- 文档怎么切
- 元数据怎么保留
- 召回结果如何过滤
- 过期文档如何清理
- 引用来源如何返回给用户
- 多租户数据如何隔离

Spring AI Alibaba 可以帮你接入向量存储和模型调用，但 RAG 的效果仍然依赖数据治理。

如果只是把一堆文档丢进向量库，最后很可能召回到过期内容、错误内容或者不该被当前用户看到的内容。

## MCP Router：把外部工具接进来

MCP Router 的意义在于把外部工具和服务发现接进 AI 应用。

当工具数量变多后，直接在应用里写死每个工具的调用方式不太现实。更好的方式是让外部能力通过标准协议暴露，再由路由层完成发现、选择和调用。

工程上要关注几件事：

- 工具是否有清晰描述
- 工具权限如何控制
- OAuth / token 如何缓存和刷新
- 调用失败如何重试
- 工具结果是否需要审计

MCP 接入后，AI 应用的边界会明显扩大。边界扩大之后，权限和审计要同步跟上。

## A2A：多 Agent 协作不要过早引入

Spring AI Alibaba 提到 A2A，也就是 Agent-to-Agent。

这个方向很有吸引力，但我不建议一开始就做复杂多 Agent。多 Agent 不是简单并行，真正难的是任务边界、状态同步、错误隔离和最终结果合并。

更稳的路线是：

1. 先把单 Agent / 单 Workflow 做稳定
2. 明确哪些能力确实需要拆出去
3. 给每个 Agent 定义输入、输出和失败语义
4. 再考虑 A2A 协作

否则很容易出现一个 Agent 的错误被另一个 Agent 放大，最后结果看起来更复杂，但并不更可靠。

## 可观测性要提前做

AI 应用如果没有日志和追踪，排查会很困难。

传统接口至少能看请求、响应、异常栈。AI 调用还需要额外关注：

- prompt
- model
- temperature
- token 消耗
- 延迟
- 工具调用参数
- 工具返回结果
- 最终输出

这些信息不一定都适合完整落库，但至少要有可追踪方案。特别是工具调用和工作流节点执行结果，不记录就很难复现问题。

## 我的使用判断

如果是 Java 团队准备做 AI 应用，我会把 Spring AI Alibaba 放在候选方案里，尤其是这些场景：

- 已经是 Spring Boot / Spring Cloud 技术栈
- 需要接企业内部工具或数据
- 需要 RAG、工具调用、工作流编排
- 希望保留 Java 后端的工程治理方式
- 后续可能做多模型、多 Agent 或可观测性

但如果只是非常轻量的聊天 Demo，直接用 OpenAI 兼容接口可能更快，不一定需要一开始就上完整框架。

## 最后

AI 应用开发的核心不是把模型调通，而是把模型调用变成可维护的工程系统。

Spring AI Alibaba 的价值也在这里：它把模型、工具、工作流、RAG、MCP 和观测能力放回 Java / Spring 的工程语境里，让后端团队可以用熟悉的方式继续演进。
