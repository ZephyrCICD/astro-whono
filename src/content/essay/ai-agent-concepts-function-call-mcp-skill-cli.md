最近接触 AI 工具越来越多，经常会看到几个词混在一起用：Function Call、MCP、Skill、Agent、CLI。

刚开始看时很容易觉得它们都差不多，都是让模型“做事”。但真放到工程里，就会发现它们其实处在不同层级。混着理解会带来一个问题：配置工具时不知道该配 MCP 还是 CLI，设计能力时不知道该写 Skill 还是直接写系统提示，讨论 Agent 时又容易把它说成一个会调用工具的聊天模型。

我现在更倾向于把它们放在一条从底层工具到上层执行的链路里看：

```text
CLI -> Function Call -> MCP -> Skill -> Agent
```

这不是严格的技术依赖关系，只是一个更容易理解的分层方式。

## CLI：具体工具的入口

CLI 是 Command Line Interface，也就是命令行工具。

它本质上是一个具体程序的操作入口，比如：

```bash
git status
python app.py
ffmpeg -i input.mp4 output.mp3
```

CLI 的特点很直接：能力强、可脚本化、适合自动化。对开发者来说，它是最常见的工具入口；对 Agent 来说，它也是很重要的执行通道。

但 CLI 本身不是 AI 协议，也不是智能体。它只负责执行明确命令，不负责理解目标、拆解任务或决定下一步。

## Function Call：模型触发工具调用的机制

Function Call 解决的是另一个问题：模型如何以结构化方式调用外部函数。

如果系统提供了这些函数：

```text
search_web(query)
read_file(path)
create_task(title, due_date)
```

当用户说“帮我查一下 MCP 的资料”时，模型可以不直接输出一段自然语言，而是返回类似这样的结构化调用：

```text
search_web(query = "MCP 最新资料")
```

Function Call 的重点不是某个具体工具，而是“模型怎么表达我要调用哪个工具、传什么参数”。它属于调用机制层，比 CLI 更抽象，也比完整 Agent 更底层。

## MCP：模型连接外部能力的标准协议

MCP 可以理解为模型连接外部资源的一套标准协议。

它主要解决的是接入方式不统一的问题。不同工具、数据库、网页抓取、文件系统、知识库都有自己的接口，如果每个 AI 工具都单独适配一遍，成本会很高。

MCP 的价值在于把这些能力包装成统一的 server，让模型或 Agent 通过一致的方式访问：

- 文件系统
- 浏览器
- 数据库
- GitHub
- 文档站点
- 内部工具

如果 CLI 更像一个具体工具入口，那么 MCP 更像一套统一插座。它不一定替代 CLI，但可以把 CLI、API、文件、数据库等能力包装成更适合模型使用的接口。

## Skill：可复用的方法和约束

Skill 不只是一个工具。它更像一份针对某类任务的操作手册。

一个好的 Skill 通常会说明：

- 什么时候应该使用
- 什么时候不应该使用
- 推荐步骤是什么
- 可能会调用哪些工具
- 输出格式有什么要求
- 哪些边界不能碰

比如“生成博客文章”可以是一个 Skill，“排查线上问题”也可以是一个 Skill。它们可能都会调用文件读取、搜索、命令行、浏览器等工具，但 Skill 本身重点不在工具，而在方法。

这也是 Skill 和 MCP 最容易混淆的地方：

- MCP 解决“工具怎么接入”
- Skill 解决“这类任务应该怎么做”

## Agent：面向目标组织执行的系统

Agent 是更上层的概念。

普通聊天模型更偏向回答问题，而 Agent 更强调完成目标。它通常会围绕一个目标循环执行：

1. 理解任务
2. 制定计划
3. 搜索和读取上下文
4. 选择工具
5. 执行操作
6. 根据结果调整
7. 交付最终结果

在这个过程中，Agent 可能会用到 Function Call、MCP、CLI、Skill，也可能使用文件系统、记忆、测试命令、浏览器等能力。

所以 Agent 不是单个工具，而是把这些能力组织起来的执行系统。

## 放到真实工作流里看

假设目标是“整理一批笔记，发布成博客文章”。

CLI 可能负责：

- 搜索文件
- 批量检查 Markdown
- 执行 Hugo 构建
- 提交 Git

Function Call 负责让模型用结构化参数调用这些能力。

MCP 可能负责连接 Obsidian、浏览器、文件系统或 GitHub。

Skill 负责约束写作流程，比如怎么清理 Obsidian 语法、怎么处理隐私信息、怎么组织 Hugo front matter。

Agent 则负责把整个任务串起来：先选题，再读取笔记，再改写，再验证，再提交。

## 最容易混淆的几个点

**Function Call 和 MCP 不是一回事。**

Function Call 是模型调用工具的机制；MCP 是模型接入外部工具和资源的协议。一个偏调用表达，一个偏连接标准。

**Skill 不等于工具。**

工具负责执行动作，Skill 负责告诉模型这类任务应该怎么做。Skill 里可能会推荐使用某些工具，但它本身不是工具集合。

**Agent 不等于“会调用工具的模型”。**

工具调用只是 Agent 的一部分。真正有用的 Agent 还需要上下文管理、执行边界、验证反馈、失败重试和最终交付。

## 总结

我现在会用这张表来记：

| 概念 | 本质 | 解决的问题 |
|---|---|---|
| CLI | 命令行工具入口 | 具体动作如何执行 |
| Function Call | 结构化调用机制 | 模型如何表达工具调用 |
| MCP | 外部能力接入协议 | 工具和数据如何统一连接 |
| Skill | 任务方法封装 | 某类任务应该如何稳定完成 |
| Agent | 目标执行系统 | 如何围绕目标规划、调用、验证和交付 |

理解这些概念后，再看 Codex、Claude Code、Cursor 或各种 Agent 框架，会更容易判断它们到底是在解决哪一层的问题。
