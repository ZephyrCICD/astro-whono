做 Agent 时很容易走向一个方向：继续加能力。

缺搜索，就接搜索；缺文件，就接文件系统；缺知识，就接 RAG；效果不稳定，就继续加系统提示词。

这些手段都有用，但它们不一定能解决真正的问题。Agent 设计里更关键的一件事，是动作空间。

动作空间可以理解为：模型在执行任务时可以采取的全部合法动作。它不只是工具列表，还包括工具边界、参数结构、错误反馈、上下文获取方式，以及模型在某个阶段能不能自然想到并正确使用这些动作。

## 工具要按模型的工作方式设计

很多工具设计一开始会站在人类工程师视角：

- 这个功能应该属于这个模块
- 这个参数人一看就懂
- 这个格式让模型遵守一下就行
- 这个错误信息开发者能看懂就够了

但模型不是人类工程师的缩小版。它不会稳定记住所有隐含约定，也不一定能从一个模糊错误里推出修复方式。

对 Agent 来说，一个好工具至少要满足几件事：

- 职责单一
- 参数结构明确
- 错误信息能指导下一步
- 返回结果足够支撑继续执行
- 工具描述能让模型在正确时机选中它

工具不是越多越好。工具越多，选择成本也越高。如果描述和边界不清楚，模型会在一堆看似相近的工具里选错。

## 高频、关键、强结构动作应该成为工具

一个典型问题是：Agent 需要向用户补充信息时，应该怎么提问？

最简单的做法是写提示词：

```text
如果你需要用户补充信息，请按固定 Markdown 格式输出问题。
```

这种方案看起来轻，但很容易不稳定。模型可能会多输出解释，也可能格式不严格，还可能把计划和问题混在一起。

如果“提问”是高频动作，而且输出结构要稳定，那它更适合作为一个正式工具，比如 `AskUserQuestion`。

这样做的好处是：

- 行为职责更清楚
- 输出结构更容易约束
- UI 可以直接接入
- 后续也更容易复用

这类判断可以迁移到很多场景：只要某个动作关键、高频、需要强结构，就不要只靠提示词硬撑。

## 不要把所有上下文提前塞给模型

代码场景里，经常会想用 RAG 先把相关上下文喂给模型。

这条路不是错的，但成本不低。索引要维护，召回要调，召回结果还可能不符合当前任务。很多时候，系统是在替模型提前判断“什么重要”。

如果模型本身已经具备不错的搜索和归纳能力，另一种方式反而更稳：给它基础探索工具，让它自己逐步找。

比如：

- 先搜索关键字
- 再读取少量文件
- 再沿引用继续追踪
- 必要时再扩大范围

这更接近熟练工程师理解陌生代码库的方式。工程师不会一上来读完整个仓库摘要，而是先找入口、看调用链、确认局部逻辑，再决定下一步。

## 主上下文要克制

Agent 很容易死于上下文膨胀。

一开始只是加一点项目说明，后来加规范、加产品文档、加工具手册、加历史经验。最后系统提示词越来越长，模型看起来“知道很多”，实际却更容易失焦。

我的理解是：主上下文应该只保留稳定且高频的信息。

适合常驻的内容包括：

- 当前目标
- 必须遵守的边界
- 项目关键约定
- 正在修改的文件
- 验证结果

不适合常驻的内容包括：

- 大段产品文档
- 低频工具说明
- 历史讨论细节
- 临时参考资料

这些内容更适合按需读取，或者交给子 Agent / Skill / 文档检索工具处理。

## 模型变强后，旧约束也要重新审视

有些约束在早期很有用，比如强制模型写 Todo、强制输出固定格式、强制每一步都解释原因。

但模型能力变强后，旧约束可能会变成负担。

比如简单 Todo 可以帮助模型保持主线，但复杂任务里它可能表达不了依赖关系、跨 Agent 状态和动态调整。此时需要的可能不是更多提示，而是把任务状态升级成可协作、可追踪的结构。

所以 Agent 设计不是一次性完成的。模型能力、工具生态、任务复杂度都会变化，动作空间也要跟着调整。

## 我现在的判断清单

设计一个 Agent 能力时，我会先问几个问题：

1. 这个动作是否高频？
2. 这个动作是否需要稳定结构？
3. 这个动作失败后，模型能不能知道怎么修？
4. 这个信息是否应该常驻上下文？
5. 这个能力是工具问题、Skill 问题，还是上下文问题？
6. 旧约束现在还是帮助，还是已经变成负担？

很多 Agent 效果差，不是模型不够强，而是动作空间让模型很难做对事。

工具、上下文、Skill、子 Agent、验证反馈，这些东西最终都在回答同一个问题：怎样给模型搭一个更适合工作的环境。
